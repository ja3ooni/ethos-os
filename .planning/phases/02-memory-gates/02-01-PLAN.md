# Plan 2.1 — Approval Gate Models

**Goal:** Define approval gate models with GateStatus enum and GateRequest model.

**Requirements covered:** GATE-01, GATE-02, GATE-03, GATE-04, GATE-05, GATE-06, GATE-07, GATE-08

## Deliverables

### 2.1.1 — GateStatus Enum
- `pending`: Gate request created, awaiting decision
- `approved`: Approver accepted the request
- `rejected`: Approver rejected the request

### 2.1.2 — GateRequest Model
- `id`: UUID primary key
- `gate_type`: scope | budget (enum)
- `entity_id`: FK to gated entity (task, project, etc.)
- `entity_type`: Type of entity (task, project, etc.)
- `status`: pending | approved | rejected
- `trigger_condition`: JSON describing why gate was triggered
- `approver`: Assigned approver (user ID)
- `notes`: Approver notes (nullable)
- `decision_notes`: Notes from approver
- `created_at`: Timestamp
- `decided_at`: Timestamp (nullable)
- `decided_by`: User ID who decided (nullable)
- `timeout_hours`: Configurable timeout

### 2.1.3 — Gate Decision Logic
- Scope gate triggers at task effort > original_estimate * 1.25
- Budget gate triggers at actual_cost > estimated_cost * 1.20
- Gate is binary (approved or rejected, no partial)

## Implementation

- Create `ethos_os/models/gate.py` with GateStatus enum and GateRequest model
- Add gate_type enum
- Add gate fields matching spec above

## Acceptance Criteria

1. GateStatus enum has pending/approved/rejected states
2. GateRequest model stores all specified fields
3. Gate type can be scoped to any entity (not just tasks)
4. Timestamp fields are auto-populated
